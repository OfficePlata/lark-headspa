# Server

バックエンド API サーバーの実装です。Express 4 + tRPC 11 で構築されており、Lark Bitable API との連携、データベース操作、認証処理を担当します。

---

## ファイル構成

| ファイル | 説明 |
|----------|------|
| `routers.ts` | tRPC ルーター定義。テーマ、サロン管理、フォーム送信の全 API を定義 |
| `db.ts` | Drizzle ORM を使用したデータベースクエリヘルパー |
| `lark.ts` | Lark Bitable API 連携ヘルパー（テナントトークン取得、レコード作成） |
| `storage.ts` | S3 ファイルストレージヘルパー |
| `salon.test.ts` | テーマ・フォーム設定・Lark フィールドマッピングのテスト |
| `auth.logout.test.ts` | ログアウト処理のテスト |
| `_core/` | フレームワーク基盤（認証、OAuth、コンテキスト等）。編集不要 |

---

## 主要モジュール

### routers.ts

tRPC ルーターを定義しています。以下の 4 つのルーターグループで構成されます。

`themes` ルーターは公開エンドポイントで、5 種類のデザインテーマの一覧取得と個別取得を提供します。`salon` ルーターは認証必須で、サロンの CRUD 操作、Lark API 設定の更新、送信履歴の取得を処理します。`form` ルーターは公開エンドポイントで、スラッグによるフォーム情報取得とフォームデータの送信を処理します。フォーム送信時には Lark Bitable への同期とオーナー通知も実行します。

### db.ts

Drizzle ORM を使用したデータベース操作関数を提供します。ユーザー、サロン、フォームフィールド、送信ログの各テーブルに対する CRUD 操作が含まれています。

### lark.ts

Lark Open Platform の API と連携するためのヘルパー関数を提供します。テナントアクセストークンの取得（`getTenantAccessToken`）と Bitable レコードの作成（`createBitableRecord`）が主な機能です。フォームデータを Lark のフィールド形式にマッピングする `mapFormDataToLarkFields` 関数も含まれています。

## Lark API 連携の仕組み

フォーム送信時の Lark Bitable 同期は以下の流れで処理されます。

1. フォームデータがクライアントから `form.submit` プロシージャに送信される
2. 送信データがローカルデータベースに保存される
3. サロンに Lark API 認証情報が設定されている場合、Lark Bitable API を呼び出す
4. テナントアクセストークンを App ID / App Secret で取得する
5. フォームデータを Lark フィールド形式にマッピングする
6. Bitable の該当テーブルにレコードを作成する
7. 同期結果（成功/失敗）をデータベースに記録する
8. オーナーに通知を送信する

Lark API の認証情報が未設定の場合でも、フォームデータはローカルデータベースに保存されます。後から Lark API を設定した場合、新しい送信分から同期が開始されます。

## テスト

```bash
pnpm test
```

`salon.test.ts` では以下をテストしています。

- 全テーマ（calmer, natural, elegant, fresh, sakura）が正しく定義されていること
- デフォルトフォーム設定が全フォームタイプに存在すること
- Lark フィールドマッピングが正しく動作すること
